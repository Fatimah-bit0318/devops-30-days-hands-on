# 🚀 Day 28 — CI/CD Pipeline with GitHub Actions, AWS ECR & Kubernetes

## 📌 Objective

The goal of Day 28 was to implement an **end-to-end CI/CD pipeline** for a containerized Flask application.

The pipeline automatically:

1. Detects code changes pushed to GitHub
2. Runs automated tests using Pytest
3. Generates application build metadata
4. Builds a Docker image
5. Pushes the Docker image to AWS ECR
6. Updates the Kubernetes deployment
7. Performs a rolling update
8. Runs the latest application version inside Kubernetes

### Final CI/CD Flow

```text
Developer
    │
    │ git push
    ▼
GitHub Repository
    │
    ▼
GitHub Actions
    │
    ▼
Automated Testing
    │
    ▼
Docker Image Build
    │
    ▼
AWS ECR
    │
    ▼
Kubernetes Deployment
    │
    ▼
Application Running
    │
    ▼
Status: Healthy ✅
```

# 1. 🐍 Flask Application

I created a Flask-based application called:

**Docker Image Information Dashboard**

The purpose of this application is to display information about the currently deployed application and Docker image.

The application provides three main endpoints:

| Endpoint | Purpose |
|---|---|
| `/` | Displays the web dashboard |
| `/info` | Returns application and deployment metadata |

The dashboard displays information such as:

```text

Build Number
Docker Image Tag

Deployment Time
Environment

Health Status
```

This makes it possible to identify exactly **which version and Git commit are currently deployed**.

# 4. 🔢 Build and Version Tracking

The GitHub Actions pipeline automatically generates information for each deployment.

For example:

```text
App Version  : 1.0.10
Build Number : 10
Image Tag    : 8481176
Git Commit   : 8481176e93715273570cfdf7b2b7b0b5a1f1ef23
Environment  : Production
```

The short Git commit SHA can be used as a unique Docker image tag.

Example:

```text
docker-image-dashboard:8481176
```

This provides **deployment traceability**.

If there is a problem with a deployment, we can identify exactly:

```text
Which build was deployed?
Which Docker image was used?
Which Git commit created it?
Which application version is running?
```

# 7. 📦 AWS ECR Integration

AWS **Elastic Container Registry (ECR)** was implemented as the private container registry for the project.

### Role of ECR

ECR stores the Docker images created by the CI/CD pipeline.

```text
GitHub Actions
      │
      ▼
Build Docker Image
      │
      ▼
Push Image
      │
      ▼
AWS ECR
      │
      ▼
Kubernetes Pulls Image
      │
      ▼
Application Runs
```

Originally, the project used Docker Hub:

```text
Docker Build
     ↓
Docker Hub
     ↓
Kubernetes
```

The final implementation uses:

```text
Docker Build
     ↓
AWS ECR
     ↓
Kubernetes
```

### Important Difference

```text
AWS ECR
   =
Stores Docker Images

Kubernetes
   =
Runs and Manages Containers
```

ECR does **not** run the application.

It stores the container image that Kubernetes uses to create the application container.

# 10. 🔒 Private ECR Authentication

The AWS ECR repository is private.

When Kubernetes first attempted to pull the ECR image, the new pod failed with:

```text
ImagePullBackOff
```

### Why did this happen?

GitHub Actions could successfully:

```text
Build Image
    ↓
Login to ECR
    ↓
Push Image ✅
```

However, Kubernetes separately needed permission to:

```text
Authenticate to ECR
    ↓
Pull Private Image
```

Without authentication:

```text
Kubernetes
     │
     ▼
Private ECR
     │
     X
Authentication Missing
     │
     ▼
ImagePullBackOff ❌
```

---

# 11. 🔑 Kubernetes ECR Secret

To allow Kubernetes to pull the private image, a Docker registry secret was created:

```text
ecr-secret
```

The Kubernetes Deployment was configured with:

```yaml
imagePullSecrets:
  - name: ecr-secret
```

The final flow became:

```text
Kubernetes
     │
     ▼
ecr-secret
     │
     ▼
Authenticate with AWS ECR
     │
     ▼
Pull Docker Image
     │
     ▼
Create Container
     │
     ▼
Pod Running ✅
```

### Important Note

AWS ECR authorization tokens are temporary.

For this learning project, an ECR pull secret was used.

In a production environment, ECR authentication should be automated using an appropriate AWS-integrated or credential-refresh mechanism.

# 12. 🔄 Kubernetes Rolling Update

When the new ECR-based image was deployed, Kubernetes performed a rolling update.

Initially:

```text
Old Pod → Running
New Pod → Starting
```

After the new pod successfully started:

```text
New Pod → 1/1 Running
Old Pod → Terminating
```

Final state:

```text
New Pod → Running ✅
Old Pod → Removed
```

### Rolling Update Flow

```text
Old Application Running
          │
          ▼
New Pod Created
          │
          ▼
New Image Pulled from ECR
          │
          ▼
New Container Started
          │
          ▼
New Pod Becomes Ready
          │
          ▼
Old Pod Terminated
```

This provides a controlled way of replacing an older application version.

# 🐛 Problems Faced and Solutions

## Problem 1 — Pytest Command Not Found

Error:

```text
pytest: command not found
```

### Solution

Pytest was installed as part of the GitHub Actions dependency installation step.

---

## Problem 2 — Python Import Error

Error:

```text
ModuleNotFoundError: No module named 'app'
```

### Solution

The tests were executed with the project root included in the Python path:

```bash
PYTHONPATH=. python3 -m pytest tests/
```

Final result:

```text
2 passed
```

After correcting the syntax:


The updated test files and workflow configuration were committed and pushed correctly.

Final result:

```text
2 passed
```

## Problem 5 — ECR Permission Error

While setting up ECR, an IAM permission error occurred:

```text

ecr:CreateRepository
```

The AWS IAM/ECR permissions were reviewed and the ECR repository was configured for the project.

## Problem 6 — Kubernetes ImagePullBackOff

Error:

```text
ImagePullBackOff
```

### Cause

Kubernetes could not authenticate with the private AWS ECR repository.

### Solution

Created:

```text
ecr-secret
```

and added:

```yaml
imagePullSecrets:
  - name: ecr-secret
```

After the fix:

```text
New Pod → 1/1 Running ✅
```

## Problem 7 — Minikube Website Access

The Minikube NodePort URL was not directly accessible from the external browser as expected because Minikube was running inside the EC2 environment.

For testing, Kubernetes port forwarding was used:

```text

      ▼
EC2 Port
      │
      ▼
kubectl port-forward
      │
      ▼
Kubernetes Service
      │
      ▼
Application Pod
```

# 🎤 Interview / Viva Questions

## What did you do on Day 28?

> I implemented an end-to-end CI/CD pipeline using GitHub Actions, Docker, AWS ECR, and Kubernetes. I configured an AWS EC2 instance as a self-hosted GitHub Actions runner. Whenever code is pushed to GitHub, the pipeline automatically runs tests, builds a versioned Docker image, pushes it to AWS ECR, and updates the Kubernetes deployment.

---

## What is CI/CD?

> CI, or Continuous Integration, automatically integrates and tests code changes. CD, or Continuous Delivery/Deployment, automates the process of delivering or deploying the tested application.

---

## What was the role of GitHub Actions?

> GitHub Actions automated the CI/CD workflow. It ran tests, built the Docker image, pushed the image to AWS ECR, and updated the Kubernetes deployment.

---

## What was the role of the self-hosted runner?

> The EC2 self-hosted runner executed the GitHub Actions jobs. It had access to Docker, AWS CLI, and Kubernetes tools required by the pipeline.

---

## What was the role of Docker?

> Docker packaged the Flask application, dependencies, runtime, and build metadata into a portable container image.

---

## What was the role of AWS ECR?

> AWS ECR was used as a private container registry to store the Docker images created by the CI/CD pipeline. Kubernetes then pulled those images from ECR for deployment.

---

## What was the role of Kubernetes?

> Kubernetes deployed and managed the application containers. It managed pods, services, image updates, and rolling deployments.

---

## What happens when you push new code?

> A push to the main branch triggers GitHub Actions. The pipeline tests the application, generates build metadata, builds a new Docker image, pushes it to AWS ECR, and updates Kubernetes with the new image.

---

## What is `ImagePullBackOff`?

> `ImagePullBackOff` means Kubernetes is unable to pull the required container image. In this project, it occurred because the AWS ECR repository was private and Kubernetes did not initially have the required authentication.

---

## How did you solve `ImagePullBackOff`?

> I created a Kubernetes Docker registry secret for ECR authentication and configured the Deployment to use it through `imagePullSecrets`.

---

## Why use a unique Docker image tag?

> A unique image tag makes every deployment traceable. In this project, the Git commit SHA was used so I could identify exactly which source-code version created a deployed image.

---

## What is a rolling update?

> A rolling update gradually replaces old application pods with new pods. Kubernetes starts the new version and, once it is ready, terminates the old version.
